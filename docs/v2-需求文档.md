## 二、详细需求

### 需求 1：登录页 + 用户认证（P0）

**参考项目**：[usememos/memos](https://github.com/usememos/memos)

memos 采用三层令牌架构（JWT Access Token + Refresh Token + PAT），我们的项目是本地单用户场景，做简化版即可。

**后端：**

1. 新增 `users` 表：
   ```sql
   CREATE TABLE users (
     id TEXT PRIMARY KEY,
     username TEXT NOT NULL UNIQUE,
     password_hash TEXT NOT NULL,  -- bcrypt 哈希
     created_at INTEGER NOT NULL
   );
   ```

2. 新增认证 API：
   | 方法 | 路径 | 说明 |
   |------|------|------|
   | POST | `/api/auth/setup` | 首次初始化（创建管理员账号，仅当无用户时可调） |
   | POST | `/api/auth/login` | 登录，返回 JWT |
   | GET | `/api/auth/status` | 检查是否已初始化（有无用户） |
   | POST | `/api/auth/logout` | 登出 |

3. JWT 中间件：所有 `/api/*` 路由（除 `/api/auth/*`）需校验 JWT Token

4. 依赖：`bcryptjs`（密码哈希）、`jsonwebtoken`（JWT 签发/验证）

**前端：**

1. 登录页（参考 memos `PasswordSignInForm.tsx` 的简洁风格）：
   - 居中卡片布局：Logo + 标题 + 用户名/密码输入框 + 登录按钮
   - 登录中显示 loading 状态
   - 错误提示（密码错误、网络异常）

2. 初始化页：
   - 首次访问时（`/api/auth/status` 返回未初始化），展示「创建管理员账号」表单
   - 设置用户名 + 密码（需确认密码）
   - 创建成功后自动登录

3. 路由守卫：
   - 未登录 → 跳转登录页
   - 已登录 → 跳转主页
   - Token 存 `localStorage`，每次请求自动带上 `Authorization: Bearer <token>`

### 需求 3：JSON 解析增强 + 兜底（P0）

**问题**：AI 返回的 JSON 经常被 markdown 代码块包裹、包含非法字符、截断等，导致解析失败，前端显示空白。

**方案**：

1. 后端 `index.ts` 中已有的 JSON 清理逻辑增强：
   - 剥离 `\`\`\`json ... \`\`\`` 代码块包装
   - 剔除中文引号 `""`
   - 尝试自动补全截断的 JSON（补括号、补引号）
   - 多轮 fallback：正则提取 → 尝试修复 → 逐字符扫描

2. 最终兜底：如果所有解析都失败，返回一个固定的「安全回复」而非报错：
   ```json
   {
     "analysis": {
       "stage": "分析中", "signal": "模糊", "strategy": "安全回复",
       "signalText": "AI 返回格式异常，已降级处理", "emotions": [],
       "tip": "建议重新生成", "favorability": 50
     },
     "plan": { "goal": "维持当前关系", "nextStep": "继续对话" },
     "replies": [{ "id": 1, "strategy": "安全回复", "text": "嗯嗯，确实", "reason": "降级兜底回复" }]
   }
   ```

3. 后端增加 LLM 调用重试：失败自动重试 1 次（同参数），重试失败再返回兜底

### 需求 4：React Error Boundary（P0）

**方案**：

1. 新建 `ErrorBoundary.tsx` 组件：
   - 捕获子组件渲染异常
   - 展示友好的错误页面：「页面出了点问题」+ 重试按钮 + 错误详情（可展开）
   - 重试按钮 → 重置 ErrorBoundary 状态

2. 在 `App.tsx` 最外层包裹 `<ErrorBoundary>`

3. 关键子组件（如 `RoundTimeline`、`ChatHistory`）单独包裹 ErrorBoundary，防止局部崩溃扩散

### 需求 5：移动端适配（P1）

**问题**：当前固定左右 70/30 布局，手机上无法使用。

**方案**：

1. 断点设计：
   - `>= 768px`：桌面端，保持左右分栏布局
   - `< 768px`：移动端，改为 Tab 切换布局

2. 移动端布局：
   ```
   ┌──────────────────────┐
   │ 顶部栏（简化）         │
   ├──────────────────────┤
   │ [聊天] [AI辅助]  Tab  │
   ├──────────────────────┤
   │                      │
   │  当前 Tab 对应内容     │
   │  (聊天 / AI辅助面板)   │
   │                      │
   ├──────────────────────┤
   │ 输入区               │
   └──────────────────────┘
   ```

3. 使用 CSS media query + 条件渲染：
   - `ChatHeader` 上新增移动端 Tab 切换按钮
   - 左右面板改为 `display: none/block` 切换
   - 回复选项弹窗在移动端改为底部全屏弹出

### 需求 6：删除操作二次确认弹窗（P1）

**问题**：删除消息目前只有一个小的删除按钮，误触风险高。

**方案**：

1. 聊天消息删除：点击删除图标 → Popconfirm 气泡确认「确定删除这条消息？」→ 确认/取消
2. 聊天对象删除：下拉列表中点击删除 → Modal 确认「删除后所有聊天记录和 AI 窗口将清除，不可恢复」→ 确认/取消
3. AI 窗口删除：已有 Popconfirm（V1 已实现），保留
4. 全部使用 antd 的 `Popconfirm` 或 `Modal.confirm`

---

### 需求 7：RoundTimeline memoization（P1）

**问题**：`RoundTimeline` 组件每次 render 都重新调用 `parseAiMessages(aiMessages)` 解析全部消息，消息多时卡顿。

**方案**：

1. 用 `useMemo` 缓存解析结果：
   ```typescript
   const rounds = useMemo(
     () => parseAiMessages(aiMessages),
     [aiMessages]
   );
   ```

2. `RoundTimeline` 组件用 `React.memo` 包裹，避免无关 state 变化导致重渲染

### 需求 8：CORS 限制 + 请求体大小限制（P0）

**后端改动：**

1. CORS 限制：
   ```typescript
   // 仅允许本地访问
   app.use(cors({
     origin: ['http://localhost:5173', 'http://localhost:5174', 'http://127.0.0.1:5173'],
     credentials: true,
   }));
   ```

2. 请求体大小限制：
   ```typescript
   app.use(express.json({ limit: '100kb' }));  // 单条消息不超过 100KB
   ```

3. API 限流（可选，用 `express-rate-limit`）：
   ```typescript
   import rateLimit from 'express-rate-limit';
   app.use(rateLimit({ windowMs: 60_000, max: 100 })); // 每分钟 100 次
   ```

### 需求 9：多模型切换（GLM / 小米 MiMo）（P1）

**目标**：用户可在前端切换 LLM 模型，不同模型有不同的 API 配置。

**后端：**

1. `.env` 新增配置：
   ```
   # 智谱 GLM
   ZHIPU_API_KEY=xxx
   ZHIPU_BASE_URL=https://open.bigmodel.cn/api/paas/v4/
   ZHIPU_MODEL=glm-5.1

   # 小米 MiMo
   MIMO_API_KEY=xxx
   MIMO_BASE_URL=https://xxx.api.xiaomi.com/v1/
   MIMO_MODEL=MiMo-7B-RL
   ```

2. `llm.ts` 改为多模型客户端管理：
   ```typescript
   const clients: Record<string, OpenAI> = {};

   function getClient(provider: string): OpenAI {
     if (!clients[provider]) {
       // 根据 provider 读取对应的 API_KEY、BASE_URL
       clients[provider] = new OpenAI({ apiKey, baseURL });
     }
     return clients[provider];
   }
   ```

3. generate 接口支持 `provider` 参数（默认 `zhipu`）

**前端：**

1. SessionBar 或 ContextBar 中新增模型选择器（Select 下拉）
2. AI 生成时将选中的模型传给后端
3. 模型列表从后端 `/api/models` 接口获取（返回已配置的可用模型）

## 五、验收标准

| 需求 | 验收方式 |
|------|---------|
| 登录认证 | 未登录访问任何页面都跳转登录页；登录后可正常使用；刷新页面不丢失登录状态 |
| 引导页 | 删除所有聊天对象后，显示引导页；创建对象后引导页消失 |
| JSON 解析 | AI 返回带 markdown 包裹 / 截断 / 乱码的 JSON 都能降级处理，不会白屏 |
| Error Boundary | 故意在组件内 throw Error，页面显示错误提示而非白屏 |
| 移动端 | iPhone SE 尺寸下可正常操作（Tab 切换聊天/AI） |
| 删除确认 | 删除消息/对象都有弹窗确认 |
| Memoization | 50+ 条 AI 消息时界面无明显卡顿 |
| CORS + 限流 | 外部域名请求被拒绝；超大请求体被拒绝 |
| 多模型 | 前端可切换 GLM/MiMo，切换后 AI 回复使用对应模型 |
