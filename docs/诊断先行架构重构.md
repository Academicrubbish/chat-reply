# 诊断先行架构重构

> 重构日期：2026-06-03
> 背景：完整模式一次 LLM 调用同时完成「分析+策略+回复」，长聊天场景下输出 JSON 过大导致截断→保底回复。军师分析产出的优质诊断结果与回复生成完全隔离，诊断价值被浪费。

---

## 核心理念

**"诊断先行，定向回复"**——诊断（军师分析）从独立功能升级为聊天目标的基础属性。回复生成**强制依赖**诊断方案，无诊断时自动先跑诊断再生成回复。

```
旧流程：对方消息 → 一次 LLM 调用（analysis + plan + replies）→ 经常截断 → 保底回复
新流程：制定方案 → 诊断（SSE 流式）→ 对方消息 → 基于诊断的精准回复（只输出 replies）
```

---

## 架构变化

### 数据模型

| 变化 | 说明 |
|------|------|
| `chat_targets.active_diagnosis_id` | 新增字段，指向当前生效的诊断（`chat_diagnoses.id`） |
| `ai_sessions.diagnosis_id` | 新增字段，记录该轮回复使用了哪个诊断 |

诊断数据复用已有的 `chat_diagnoses` 表，不需要新建表。

### 新增 API

| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/targets/:targetId/diagnose` | POST | SSE 流式诊断，成功后自动设为 active_diagnosis |
| `/api/targets/:targetId/active-diagnosis` | GET | 获取当前生效的诊断 |
| `/api/targets/:targetId/active-diagnosis` | DELETE | 清除诊断 |

### 改造 API

| 端点 | 变化 |
|------|------|
| `POST /api/sessions/:id/generate` | 无诊断时自动调用 `runDiagnosisInternal`（非流式），然后基于诊断生成回复；有诊断时直接用。输出简化为只返回 `replies`（不再要求 analysis/plan）。SSE 新增 `auto_diagnosing` 和 `diagnosis_ready` 事件。 |
| `POST /api/sessions/:id/regenerate` | 同上，加入自动诊断逻辑。 |

### Prompt 变化

| 维度 | 旧 | 新 |
|------|-----|-----|
| 知识单元 | 关键词匹配全量 23 单元 | 诊断指定 `knowledgeIds`（通常 3-8 个） |
| 上下文注入 | 无诊断摘要 | 注入完整诊断块（阶段/态度/情绪/策略/警告） |
| 输出格式 | `{analysis, plan, replies}` | `{replies}` （轻 40%+） |
| Token 压力 | 高（3000-7000 输入 + 800-1200 输出） | 低（1800-4000 输入 + 300-500 输出） |

### 前端变化

| 组件 | 变化 |
|------|------|
| `Toolbar` | 新增「制定方案/重新诊断」按钮 + 诊断状态 Tag |
| `useAppState` | 新增 `activeDiagnosis`/`isDiagnosing`/`diagnosisStep` 状态，6 个新 action，`diagnoseTarget`/`clearDiagnosis` 函数 |
| `types` | 新增 `TargetDiagnosis` 接口 |
| `api` | 新增 `diagnoseStream`/`getActiveDiagnosis`/`clearActiveDiagnosis` |

## 用户体验变化

1. **新建 target + 导入聊天** → Toolbar 显示蓝色「制定方案」按钮
2. **点击「制定方案」** → SSE 流式诊断 → 完成后 Toolbar 显示诊断 Tag（如 `认识→交流 · 思维匹配`）
3. **直接点 AI 辅助（无诊断）** → 后端自动先跑诊断（步骤条显示"自动诊断中"），然后生成回复，用户无需手动操作
4. **有诊断后点 AI 辅助** → 直接生成回复，基于诊断方向，输出更精准
5. **切换 target** → 自动加载该 target 的诊断
6. **「重新诊断」** → 更新诊断方案，后续回复跟随新方向

---

## 向后兼容

- 数据库迁移全部是 `ALTER TABLE ADD COLUMN` + `DEFAULT NULL`，不破坏现有数据
- 已有 `chat_diagnoses` 记录会被自动激活为该 target 的 active diagnosis
- 旧版前端不会崩溃——只是永远不调用新 API，后端 generate 在无诊断时自动处理
